## A. スキャン方向・タイミング系

### A-1. 窓左端スキャンバグ（#021 → #022で修正）

**症状**: 全13件のエントリーが「底を打つ前」に発火していた

**原因**:
`scan_window_entry()` が窓の先頭（1H SL より20時間前）から
5Mバーを順にスキャンし、最初の「min(open,close) > neck_15m」を拾っていた。
これは下落途中でまだネック上にいる足であり、底形成後の上抜けではなかった。

```
価格
 ↑  ①誤検出（下落前）    ③正解（底の後）
 |  ＼  neck -------＼---/------
 |    ＼           ＼ /
 |     ＼___底(SL)__/
 +--------------------------------→ 時間
      窓の左端    1H SL    窓の右端
```

**修正（3行）**:
```python
sl_1h_idx = df_5m_win.index.searchsorted(sl_1h_ts)
for j in range(sl_1h_idx, len(df_5m_win) - 1):
```

**教訓**: 時間窓スキャンでは「どの時点以降からスキャンするか」を必ず明示する。
窓全体からパターンを検出するのは正しいが、エントリー判定は底確定後に限定すべき。

---

### A-2. 1H SL 一致検証で別の波の底を比較（#020旧版 → #020v2で修正）

**症状**: 4H SL と 1H SL の距離が median 206 pips と表示された

**原因**:
`get_all_swing_lows_1h(lookback=240).min()` は10日間の**絶対最安値**を返す。
4H SL が「3日前の押し目底 = 157.5」なのに、1H側は「8日前の大底 = 154.0」を返し、
**別の波の底**同士を比較していた。

**修正**:
4H SL のタイムスタンプ ±8本（8時間）窓内で**最近傍**の1H SLを探す方式に変更。
→ 結果: 89件100%一致（距離0.0pips）

**教訓**: 「最安値」と「同じ構造の安値」は全く別の概念。
時間的近傍で探さないと、異なる相場局面の値を比較してしまう。

---

### A-3. Support_1h の lookback が短すぎた（#020で発覚・#021で転換）

**症状**: 旧版の Support_1h が不正確で、4H構造と噛み合わない

**原因**:
`get_nearest_swing_low_15m(n=3, lookback=20)` → 直近20本 = 5時間分。
4H足の押し目波が形成されるには約1週間かかる。5時間の窓では
4H構造の押し目底を捉えられず、構造的にミスマッチしていた。

**修正**:
3層階層スキャンに転換。4H SL → 1H窓 → 窓内15M/5Mの
トップダウン構造に変更し、lookback依存の問題を根本解消。

**教訓**: lookback の長さは「検出したい構造の形成期間」に合わせる必要がある。
下位足の短い lookback で上位足の構造を捉えようとすると構造的に破綻する。

## B. API シグネチャ不一致系

### B-1. get_direction_4h の current_idx 問題（#020）

**症状**: ClaudeCode が指示書通りに呼び出すと TypeError

**原因**:
指示書では `get_direction_4h(high, low, n=3, lookback=20)` と想定していたが、
実際は `get_direction_4h(high, low, current_idx, n=3, lookback=20)` で
`current_idx` が必須引数だった。

**修正**: ClaudeCode が実ファイルを read して正しいシグネチャで呼び出し。

**教訓**: 指示書のコードは「想定」であり、実際のAPIと一致しない可能性が常にある。
ClaudeCode には必ず「実ファイルの def 行を read してから呼び出すこと」と指示する。

---

### B-2. check_15m_range_low の返り値型（#021）

**症状**: `result[0]`, `result[1]` でアクセスすると KeyError

**原因**:
指示書では `→ (pattern, neck_15m, details)` という tuple を想定していたが、
実際は `→ dict` で `result['found']`, `result['pattern']`, `result['neck_15m']` だった。
さらに `direction` 引数が必須（デフォルトなし）だった。

**修正**: dict アクセスに変更 + `direction='LONG'` を追加

**教訓**: B-1と同じパターン。特に返り値の型（tuple vs dict vs namedtuple）は
ドキュメントに書かれていないことが多く、実コード確認が必須。
**指示書に「シグネチャ確認手順」を毎回含めること。**

---

### B-3. 指示書に旧版と修正版が混在（#020指示書）

**症状**: ClaudeCode が旧版（バグあり）を実装してしまうリスク

**原因**:
指示書の構造が「前半: 旧版コード → 中間: バグ報告 → 後半: 修正版コード」
になっていた。上から読んで旧版を実装する危険性があった。

**修正**: 旧版を完全削除し、修正版のみの一本化ファイルとして再発行。

**教訓**: 指示書には「最終版のコードのみ」を含める。
バグ修正の経緯は別セクション（「修正履歴」など）に記載し、
実装すべきコードブロックを1つだけにする。

## C. mplfinance 固有の罠

### C-1. make_addplot() に ax= 引数は存在しない（#020）

**症状**: TypeError

**原因**: matplotlib の標準APIと異なり、mplfinance の `make_addplot()` は
`ax=` パラメータを受け付けない。

**修正**: `ax=` を削除。addplot のターゲット軸は `mpf.plot()` 側で制御する。

---

### C-2. addplot が axes[1] 白帯を生成する（#020-fix）

**症状**: ローソク足が表示されない（データは存在するがチャートエリアが空白）

**原因**: `mpf.make_addplot()` を使うと mplfinance が内部的に `axes[1]` パネルを
生成し、図の22%を白帯で占有する。ローソク足は axes[0] に描画されているが、
白帯に覆われて見えなくなっていた。

**修正**: addplot を完全廃止 → `ax.scatter()` で整数x軸に直接描画。

---

### C-3. axvline に Timestamp を渡しても表示されない場合がある（#020）

**症状**: 垂直線が表示されない

**原因**: mplfinance の `returnfig=True` モードでは x軸が内部的に
整数インデックスに変換される場合がある。Timestamp 渡しでは位置が合わない。

**修正**: try/except フォールバック。Timestamp で失敗したら整数インデックスで再試行。

---

### C-4. CJK文字が豆腐になる（#020）

**症状**: タイトルの日本語が □□□ と表示される

**原因**: mplfinance のデフォルトフォント DejaVu Sans に CJK 文字がない。

**修正**: タイトルを英語に変更。

**C系の統合教訓（不変ルール）**:
```
1. returnfig=True パターンのみ使う
2. make_addplot() に ax= を渡さない
3. addplot の代わりに ax.scatter() で描画する
4. axvline は try/except でフォールバック
5. タイトルに日本語を使わない
```

## D. パラメータ設計ミス

### D-1. ネック許容をPCT（%）で計算してしまった（#017で修正）

**症状**: NECK_TOLERANCE が ±450pips になり、事実上フィルターとして機能しない

**原因**: `NECK_TOLERANCE_PCT = 0.03`（3%）を USDJPY 150円台に適用すると
150 × 0.03 = 4.5円 = 450pips。為替の価格水準が高いため、%ベースだと
許容範囲が異常に広くなる。

**修正**: `NECK_TOLERANCE_PIPS = 20.0` に変更（pip固定値）

**教訓**: USDJPY のような価格水準が高い通貨ペアでは、
**全てのトレランスをpip（絶対値）ベースにする**。PCT ベースは使わない。

---

### D-2. MIN_4H_SWING_PIPS ガードが無かった（#010で追加）

**症状**: 極端に狭い4Hスイング幅でFibonacciフィルターをバイパスしてしまう

**原因**: 4H SH - SL = 5pips のような狭いレンジでは、
Fib 61.8% 圏内に大部分の価格が入ってしまい、フィルターが無意味になる。

**修正**: `MIN_4H_SWING_PIPS = 20` を追加。スイング幅20pips未満はスキップ。

---

### D-3. 4H SH/SL = 0本の時のフォールバック値問題（#012以前）

**症状**: Swing が検出されない期間でもエントリーが発火する

**原因**: SH/SL が 0本の時にフォールバック値（直近high/low）が返され、
スイング幅ガードを常にパスしてしまっていた。

**修正**: 「SH/SL count = 0 の場合はスキップ」ガードを追加。

---

### D-4. WINDOW_1H_POST が短すぎた（#022 → #023で修正）

**症状**: #022修正後、エントリーが2件（IHS×2のみ）に激減

**原因**: `WINDOW_1H_POST = 5`（底の後5時間）では、
DB の2番底形成やASCENDINGの安値切り上げ完了を見逃していた。
15M パターンが完成するには最低6時間（15M × 6本 × 4サイクル）必要。

**修正**: `WINDOW_1H_POST = 5 → 10`（後10時間）に延長
→ 2件 → 5件（DB:2 / IHS:1 / ASCENDING:2）

**教訓**: 窓の大きさは「検出したいパターンの形成に必要な最小時間」
から逆算して決める。15M DBに6時間かかるなら、窓の右側は最低6時間以上必要。

## E. アーキテクチャ判断（却下した選択肢と理由）

### E-1. ボトムアップ走査 → トップダウン3層スキャンへの転換（#020）

**旧設計**: 5Mバーを1本ずつ全走査（83,112本）→ 各バーで4H/1H/15M条件を判定

**問題**: Support_1h = 15M直近20本（5時間）で計算。
4H押し目波の形成に1週間かかるのに、5時間では構造を捕捉できない。

**新設計（採用）**:
```
Layer 1: 4H → 上昇ダウ確認 → 各4H SL ts取得
Layer 2: 1H → SL付近の窓確定（前20+SL+後10=31本）
Layer 3: 窓内 → 15M DB/IHS → 5Mネック越え → エントリー
```

**却下理由**: ボトムアップだと上位足の構造を短い lookback で近似するしかなく、
構造的ミスマッチが避けられない。

---

### E-2. SHORT 運用の一時停止（#012）

**旧設計**: LONG + SHORT 両方向を同時運用

**問題**: SHORT トレードが構造的にアンダーパフォーム。
PF 0.98 → LONG限定にしたら PF 2.86 に改善。

**判断**: `DIRECTION_MODE = 'LONG'` に固定。SHORT は将来サンプル充足後に再検討。

**教訓**: データが示す非対称性には素直に従う。
「理論的にはSHORTも動くはず」ではなく、実データのエビデンスで判断する。

---

### E-3. Phase 1（シンプル版）を先に実装する判断（#021）

**選択肢A（採用）**: 15M ネック越え → 5M実体確定（1段）
**選択肢B（却下）**: 15M右肩内5M DB ネック越え（2段ネスト）

**理由**:
1. バグの切り分けが容易（検証対象が1点のみ）
2. シンプル版の結果がフル版の期待値ベースラインになる
3. 既存 check_15m_range_low() をそのまま流用でき実装コスト1/3

**結果**: Phase 1で #021のタイミングバグ（A-1）を発見できた。
フル版で同時に実装していたら、バグの原因切り分けがはるかに困難だった。

## ClaudeCode 向け不変ルール（全指示書共通）

以下は過去のバグから導出された、全指示書に適用する不変ルール:

```
1. 既存関数を呼ぶ前に必ず実ファイルの def 行を read する（B-1, B-2）
2. backtest.py / entry_logic.py / exit_logic.py / swing_detector.py は
   明示的に許可されない限り変更しない
3. resample_tf は label='right', closed='right' で統一。変更禁止
4. mplfinance は returnfig=True パターンのみ（C系全体）
5. 全てのトレランスは pip ベース。PCT ベースは使わない（D-1）
6. 指示書のコードをそのままコピペせず、実APIに合わせる（B-1, B-2）
7. git diff で禁止ファイルの差分ゼロを完了条件として確認する
8. エラーが出たら自分で「想像で」修正しない。Rex に報告して停止
```
