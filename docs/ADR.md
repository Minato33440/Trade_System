## このファイルの目的

新スレッドを立てるたびに、設計者（Rex-Planner / Rex-Evaluator）は
コンテキストが薄くなり、過去に踏んだ地雷を再び踏むリスクがある。
本ファイルは3つの機能を持つ:

1. **失敗パターン集（A〜D章）**: 何が失敗して何で直したか
2. **アーキテクチャ判断記録（E章）**: なぜその設計にしたか、却下した選択肢
3. **設計方針ガイド（F章）**: 今後の実装で迷った時の判断基準

**Plannerへ**: 設計判断で迷ったらまずF章を確認すること。
F章の方針に沿った選択肢を採用し、逸脱する場合はEvaluatorに確認を取ること。

## A. スキャン方向・タイミング系

### A-1. 窓左端スキャンバグ（#021 → #022で修正）

**症状**: 全13件のエントリーが「底を打つ前」に発火していた

**原因**: `scan_window_entry()` が窓の先頭（1H SL より20時間前）から
5Mバーを順にスキャンし、最初の「min(open,close) > neck_15m」を拾っていた。
底形成後の上抜けではなく下落途中の残存足だった。

**修正（3行）**:
```python
sl_1h_idx = df_5m_win.index.searchsorted(sl_1h_ts)
for j in range(sl_1h_idx, len(df_5m_win) - 1):
```

**教訓**: 時間窓スキャンでは「どの時点以降からスキャンするか」を必ず明示する。

---

### A-2. 1H SL 一致検証で別の波の底を比較（#020旧版 → #020v2で修正）

**症状**: 4H SL と 1H SL の距離が median 206 pips と表示された

**原因**: `get_all_swing_lows_1h(lookback=240).min()` は10日間の絶対最安値を返す。
別の波の底同士を比較していた。

**修正**: 4H SL のタイムスタンプ ±8本窓内で最近傍の1H SLを探す方式に変更。
→ 結果: 89件100%一致（距離0.0pips）

**教訓**: 「最安値」と「同じ構造の安値」は全く別の概念。

---

### A-3. Support_1h の lookback が短すぎた（#020で発覚・#021で転換）

**症状**: 旧版の Support_1h が不正確で、4H構造と噛み合わない

**原因**: `get_nearest_swing_low_15m(n=3, lookback=20)` → 直近20本 = 5時間分。
4H足の押し目波が形成されるには約1週間かかる。

**修正**: 3層階層スキャンに転換（4H SL → 1H窓 → 窓内15M/5M）

**教訓**: lookback の長さは「検出したい構造の形成期間」に合わせる。

---

### A-4. 1H SL 選択の後期バイアス（#024a分析 → #025で解決）

**症状**: 再テスト（6時間後）が初回タッチより優先された

**修正**: 価格許容範囲（20pips）内で「最も早い」1H SLを優先する方式に変更。

**教訓**: 押し目底は初回タッチが本来の機会。再テストは追認にすぎない。

---

### A-5. neck 選択の後期バイアス（#024a分析 → #025で解決）

**症状**: `sh_vals.max()` が本来のneckより上のレンジ高値を選択

**修正**: `sh_vals.iloc[0]`（固定ネック原則）→ さらに #026a で統一neck原則に転換

**結果**: 4件 → 15件（+275%）

**教訓**: 最初の抵抗線がneck。ブレイク後の高値はもはやneckではない。

### D-6. 決済トリガーの認識誤り（#026a設計時 → 2026-04-14修正）

**正しい定義（ボス確認済み）**:

neck_1h = 窓特定のアンカー（決済トリガーではない）
```

**教訓**: 決済ロジックの段階定義は EX_DESIGN の 3-5 節が唯一の正。

---

### D-7. 1H Swing n=2 が細かすぎる（#026a-verify → n=3 に変更確定）

**修正**: 1H n=2 → n=3（前後3本 = 7時間幅）。TVチャートとの整合性合格。

---

### D-8. exit_logic.py 方式B迂回（#026b実装時・2026-04-17確定）

**症状**: ClaudeCode が manage_exit() の呼び出しを回避し、独自の決済ロジック（方式B）を実装した。

**原因**: exit_logic.py（凍結ファイル）内の manage_exit() は position dict の neck_1h を
半値決済トリガーとして参照する設計。ADR D-6 では neck_4h が正しい半値決済トリガー。

**対応**: exit_simulator.py の方式B（独自実装）を正式な決済エンジンとして採用。
exit_logic.py の manage_exit() は旧版として凍結保持。

**教訓**: 凍結ファイルの内部ロジックが ADR の最新定義と乖離する場合、
凍結を解除するのではなく、新しいファイルに正しいロジックを実装する。
凍結の目的は「ベースライン再現性」であり、ロジックの正しさではない。

---

### D-9. WICKTOL_PIPS → ENTRY_OFFSET_PIPS 置換（#026c・2026-04-17確定）

**症状**: #026b結果で stage2 breakeven_stop の4件全てがマイナスP&L。
entry_price が neck_4h を超過（例: #04 neck=151.790 / entry=152.651 / +86.1pips離れ）。

**旧パラメータ**: WICKTOL_PIPS=5.0 / min(open,close) > neck+5pips / 次足始値
**新パラメータ**: ENTRY_OFFSET_PIPS=7.0 / bar.high >= neck+7pips / 固定価格

**教訓**: エントリー価格が構造的な利確ライン（neck_4h）を超えうる方式は、
決済段階で破綻する。指値方式でエントリー位置を固定すべき。

---

### D-10. 4H構造優位性フィルター（#026d・2026-04-17確定）

**症状**: #026c #07 で stage2_breakeven_stop PnL=-13.10。
neck_4h(157.960) < entry_price(158.222)。4H neckブレイク後の裏確認パターン。

**修正**: `neck_4h < neck_1h` の場合はエントリー除外。
数学的根拠: sl_4h ≈ sl_1h（#020検証済み）なので neck_4h >= neck_1h に等価。

**#026d結果**: 13件→10件（3件除外・全て負けトレード・誤除外0件）
PF: 2.42→4.54 / 勝率: 46.2%→60.0% / MaxDD: 69.4p→35.8p / 総損益: +113.3p→+150.6p

**教訓**: neck_4h が半値決済として機能するには neck_4h >= neck_1h が前提。
ADR F-1（トップダウン原則）のエントリー検証への直接応用。パラメータゼロ。

### E-6. neck_1h / neck_4h の取得方法 — CSV追加 vs exit内再計算（#026設計判断）

### E-7. エントリー方式の転換 — 実体越え次足始値 → 指値到達方式（#026c・2026-04-17確定）

**選択肢A（却下）**: entry_price に上限（MAX_ENTRY_SPREAD_PIPS）→ 機会損失
**選択肢B（却下）**: neck_1h 手前をエントリー条件 → neck_1h精度未検証
**選択肢C（採用）**: 指値方式（neck_15m + ENTRY_OFFSET_PIPS=7.0）

**採用理由（ボス判断・2026-04-15）**:
1. 損切幅の安定化（初動SL=15Mダウ崩れとの整合）
2. 指値注文との等価性（裁量での再現可能）
3. バックテスト再現性（一意の価格）
4. 構造的レイトエントリー防止

**エントリー判定方式の変遷**:
```
#025:  sh_vals.iloc[0]（SL以降初回SH）

#026c: 統一neck + 7pips 指値（構造的位置固定）

## F. 設計方針ガイド（Planner必読）

**本章は Evaluator（Opus）が発行・管理する。**

---

### F-1. データの流れ: 上位足 → 下位足（トップダウン原則）

```
確定済みの3層構造:
  Layer 1（4H）→ Layer 2（1H窓）→ Layer 3（15M/5M窓内）

全ての情報は上位足から下位足へ渡す。
下位足が上位足の情報を「逆算」や「再計算」してはならない。
```

---

### F-2. エントリー文脈はエントリー時に確定する

```
確定すべきエントリー文脈:
  ✅ entry_price, entry_ts        （エントリー自体）
  ✅ sl_4h, neck_4h               （4H構造 = 損切基準 / 半値決済トリガー）
  ✅ neck_15m                     （15Mエントリートリガー）
  ✅ neck_1h                      （窓特定のアンカー）
  ✅ pattern                      （DB/IHS/ASCENDING）
  ✅ sl_1h_ts, sl_4h_ts           （構造の時間的位置）

決済ロジックは CSV から読み取るだけ。再計算しない。
```

---

### F-3. 関数の責務を単一にする（用途限定の原則）

```
確定済みの用途限定:
  check_15m_range_low() → パターンラベル取得のみ
  neck計算             → SL以前の最後のSH（統一neck原則）
  manage_exit()        → 使用停止（D-8: 方式B迂回）
  exit_simulator.py    → 方式B: 正式な決済エンジン
  scan_4h_events()     → 4H構造スキャン（SH/SLペアで返す）
```

### F-5. 設計判断で迷った時の優先順位

```
1. 上位足構造の整合性 > 実装の簡便さ
2. エントリー時確定 > 決済時再計算
3. シンプル版を先に > フル版を一気に
4. バグ切り分け可能 > 機能の網羅性
5. 過去の失敗パターン（A〜D章）に該当しないか確認
```

---

### F-6. 各タイムフレーム SH/SL の目的定義（2026-04-17 更新）

```
■ 4H SH/SL
  4H SH = neck_4h = 半値決済トリガー（段階2）
  4H SL = 押し目候補の起点

■ 1H SH/SL
  1H SH = neck_1h = 窓特定アンカー + 4H構造優位性フィルター基準値
  1H SL = 4H SLと一致する転換水準

■ 15M SH/SL
  15M SH = neck_15m = エントリートリガー（neck+7pips指値）
  15M SL = DB/IHS/ASCENDING パターン構成要素

■ 5M SH/SL
  5M = 決済時のダウ崩れ判定

統一neck原則（全TF共通）:
  neck = sh_before_sl.iloc[-1]  # SL直前の最後のSH

Swingパラメータ確定値:
  4H: n=3 / 1H: n=3 / 15M: n=3 / 5M: n=2

4H構造優位性フィルター（#026d確定）:
  neck_4h >= neck_1h（エントリー有効の必要条件）
```
