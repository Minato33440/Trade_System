## A. スキャン方向・タイミング系

### A-1. 窓左端スキャンバグ（#021 → #022で修正）

**症状**: 全13件のエントリーが「底を打つ前」に発火していた

**原因**: `scan_window_entry()` が窓の先頭（1H SL より20時間前）から
5Mバーを順にスキャンし、最初の「min(open,close) > neck_15m」を拾っていた。
底形成後の上抜けではなく下落途中の残存足だった。

**修正（3行）**:
```python
sl_1h_idx = df_5m_win.index.searchsorted(sl_1h_ts)
for j in range(sl_1h_idx, len(df_5m_win) - 1):
```

**教訓**: 時間窓スキャンでは「どの時点以降からスキャンするか」を必ず明示する。

---

### A-2. 1H SL 一致検証で別の波の底を比較（#020旧版 → #020v2で修正）

**症状**: 4H SL と 1H SL の距離が median 206 pips と表示された

**原因**: `get_all_swing_lows_1h(lookback=240).min()` は10日間の絶対最安値を返す。
別の波の底同士を比較していた。

**修正**: 4H SL のタイムスタンプ ±8本窓内で最近傍の1H SLを探す方式に変更。
→ 結果: 89件100%一致（距離0.0pips）

**教訓**: 「最安値」と「同じ構造の安値」は全く別の概念。

---

### A-3. Support_1h の lookback が短すぎた（#020で発覚・#021で転換）

**症状**: 旧版の Support_1h が不正確で、4H構造と噛み合わない

**原因**: `get_nearest_swing_low_15m(n=3, lookback=20)` → 直近20本 = 5時間分。
4H足の押し目波が形成されるには約1週間かかる。

**修正**: 3層階層スキャンに転換（4H SL → 1H窓 → 窓内15M/5M）

**教訓**: lookback の長さは「検出したい構造の形成期間」に合わせる。

---

### A-4. 1H SL 選択の後期バイアス（#024a分析 → #025で解決）

**症状**: 再テスト（6時間後）が初回タッチより優先された

**修正**: 価格許容範囲（20pips）内で「最も早い」1H SLを優先する方式に変更。

**教訓**: 押し目底は初回タッチが本来の機会。再テストは追認にすぎない。

---

### A-5. neck 選択の後期バイアス（#024a分析 → #025で解決）

**症状**: `sh_vals.max()` が本来のneckより上のレンジ高値を選択

**修正**: `sh_vals.iloc[0]`（固定ネック原則）→ さらに #026a で統一neck原則に転換

**結果**: 4件 → 15件（+275%）

**教訓**: 最初の抵抗線がneck。ブレイク後の高値はもはやneckではない。

### D-6. 決済トリガーの認識誤り（#026a設計時 → 2026-04-14修正）

**正しい定義（ボス確認済み）**:
```

neck_1h = 窓特定のアンカー（決済トリガーではない）
```

**教訓**: 決済ロジックの段階定義は EX_DESIGN の 3-5 節が唯一の正。

### D-7. 1H Swing n=2 が細かすぎる（#026a-verify → n=3 に変更確定）

**修正**: 1H n=2 → n=3（前後3本 = 7時間幅）。TVチャートとの整合性合格。

### D-8. exit_logic.py 方式B迂回（#026b実装時・2026-04-17確定）

**症状**: ClaudeCode が manage_exit() の呼び出しを回避し、独自の決済ロジック（方式B）を実装した。

**原因**: exit_logic.py（凍結ファイル）内の manage_exit() は position dict の neck_1h を
半値決済トリガーとして参照する設計。ADR D-6 では neck_4h が正しい半値決済トリガー。

**対応**: exit_simulator.py の方式B（独自実装）を正式な決済エンジンとして採用。
exit_logic.py の manage_exit() は旧版として凍結保持。

**教訓**: 凍結ファイルの内部ロジックが ADR の最新定義と乖離する場合、
凍結を解除するのではなく、新しいファイルに正しいロジックを実装する。
凍結の目的は「ベースライン再現性」であり、ロジックの正しさではない。

---

### D-9. WICKTOL_PIPS → ENTRY_OFFSET_PIPS 置換（#026c・2026-04-17確定）

**症状**: #026b結果で stage2 breakeven_stop の4件全てがマイナスP&L。
entry_price が neck_4h を超過（例: #04 neck=151.790 / entry=152.651 / +86.1pips離れ）。

**旧パラメータ**: WICKTOL_PIPS=5.0 / min(open,close) > neck+5pips / 次足始値
**新パラメータ**: ENTRY_OFFSET_PIPS=7.0 / bar.high >= neck+7pips / 固定価格

**教訓**: エントリー価格が構造的な利確ライン（neck_4h）を超えうる方式は、
決済段階で破綻する。指値方式でエントリー位置を固定すべき。

---

### D-10. 4H構造優位性フィルター（#026d・2026-04-17確定）

**症状**: #026c #07 で stage2_breakeven_stop PnL=-13.10。
neck_4h(157.960) < entry_price(158.222)。4H neckブレイク後の裏確認パターン。

**修正**: `neck_4h < neck_1h` の場合はエントリー除外。
数学的根拠: sl_4h ≈ sl_1h（#020検証済み）なので neck_4h >= neck_1h に等価。

**#026d結果**: 13件→10件（3件除外・全て負けトレード・誤除外0件）
PF: 2.42→4.54 / 勝率: 46.2%→60.0% / MaxDD: 69.4p→35.8p / 総損益: +113.3p→+150.6p

**教訓**: neck_4h が半値決済として機能するには neck_4h >= neck_1h が前提。
ADR F-1（トップダウン原則）のエントリー検証への直接応用。パラメータゼロ。

**裁量思想との整合**（2026-04-19 MTF_INTEGRITY_QA Q3 で確認）:
D-10 は「エンジニアリング発見の後付け」ではなく、裁量思想のフラクタル構造から
**必然的に導かれる構造制約**である。ボス回答: 「エリオット波動をフラクタル構造で
見れば、neck_4h ＜ neck_1h ということは 1h上昇トレンド中になるはず」。
D-10 は (1) 構造機能：フラクタル的にあり得ない状態の除外 + (2) 防御機能：ボラ急増時の
リスクヘッジ の 2 機能を同時に果たす。

### D-12. stage2 建値移動の 🤖 創作混入確定（2026-04-19 確定・認識の固定）

**症状**: `exit_simulator.py` の `stage2_breakeven_stop` 判定に「4時間ネック到達・
半値決済 + **残り 50% のストップを建値移動**（ノーリスク化）」ロジックが実装されている。
しかし `MINATO_MTF_PHILOSOPHY.md §3 段階③` は「4時間ネック到達 → 半値決済」とのみ
記述され、建値移動は本来の裁量思想に存在しない。

**ボス証言**（MTF_INTEGRITY_QA Q6・2026-04-19）:

> 当初私が 15M 2波移行は 4H-SwgH 到達までは 5Mダウ崩れ決済と定義していたが、
> テストPlot で 4H到達半値決済後の 4H neck 越え 15Mダウ崩れ決済が一度も
> 特定されなかったので、**確かバグ特定のために仮に置いたものだと思うが**。

**判定**: 🤖 **創作混入確定**。実装時のデバッグ/テスト用の仮設置が、そのまま本採用
された経緯と推察される。

**実害評価**:
- ノーリスク化自体は多くのトレーダーが実践する合理的挙動
- 現 #026d の PF 4.54 は本ロジック込みの結果
- 即訂正は #026d の静的点を動かすことになる

**対応**（本 ADR の役割は認識の固定）:
- 本 ADR で「裁量思想にない創作である」ことを確定・記録する
- 実装訂正は **Phase 4（REX_029 以降）**。Planner が訂正指示書を起草、Evaluator
  承認、ClaudeCode 実装、新 PF を静的点として記録
- 裁量整合版では段階③を「4時間ネック到達 → 半値決済」のみに簡素化する

**教訓**:
- デバッグ用の仮設置が本採用されるリスク
- 実装時には「なぜこのロジックが必要か」を裁量思想とマッピングしてから確定する
  習慣が必要
- **原則α（シンプルな土台の保守）違反の典型例**

**関連**: MTF_INTEGRITY_QA.md 2026-04-19 Q6 / ADR F-8 原則α / ADR D-13（兄弟事例）

### E-6. neck_1h / neck_4h の取得方法 — CSV追加 vs exit内再計算（#026設計判断）

**選択肢B（採用）**: window_scanner.py に sh_4h / neck_1h カラムを追加。
理由: DRY原則、エントリー文脈のエントリー時確定、exit_simulatorの自己完結性。

**半値決済の執行価格**: α方式（neck_4h 固定値）を採用。

**各neckの用途定義（2026-04-14 ボス確認済み）**:
```
neck_15m — エントリートリガー（5M high >= neck+7pips で指値約定）
neck_1h  — 窓特定のアンカー + 4H構造優位性フィルター基準値
neck_4h  — 半値決済トリガー（段階2: High >= neck_4h → 50%決済）
```

### E-7. エントリー方式の転換 — 実体越え次足始値 → 指値到達方式（#026c・2026-04-17確定）

**選択肢A（却下）**: entry_price に上限（MAX_ENTRY_SPREAD_PIPS）→ 機会損失
**選択肢B（却下）**: neck_1h 手前をエントリー条件 → neck_1h精度未検証
**選択肢C（採用）**: 指値方式（neck_15m + ENTRY_OFFSET_PIPS=7.0）

**採用理由（ボス判断・2026-04-15）**:
1. 損切幅の安定化（初動SL=15Mダウ崩れとの整合）
2. 指値注文との等価性（裁量での再現可能）
3. バックテスト再現性（一意の価格）
4. 構造的レイトエントリー防止

**エントリー判定方式の変遷**:
```
#025:  sh_vals.iloc[0]（SL以降初回SH）
#026a: sh_before_sl.iloc[-1]（統一neck・SL直前最後SH）
#026c: 統一neck + 7pips 指値（構造的位置固定）
```

**暫定性の認識**（2026-04-19 MTF_INTEGRITY_QA Q5 で確認）: 
ENTRY_OFFSET_PIPS=7.0 はボスが明確に「仮設定」と認識しており、将来のボラティリティ
係数（STEP⑥）実装時に動的化する意図が裁量思想に組み込まれている。🟡 暫定パラメータ。

### F-6. 各タイムフレーム SH/SL の目的定義（2026-04-17 更新）

```
■ 4H SH/SL
  4H SH = neck_4h = 半値決済トリガー（段階2）
  4H SL = 押し目候補の起点

■ 1H SH/SL
  1H SH = neck_1h = 窓特定アンカー + 4H構造優位性フィルター基準値
  1H SL = 4H SLと一致する転換水準

■ 15M SH/SL
  15M SH = neck_15m = エントリートリガー（neck+7pips指値）
  15M SL = DB/IHS/ASCENDING パターン構成要素

■ 5M SH/SL
  5M = 決済時のダウ崩れ判定

統一neck原則（全TF共通）:
  neck = sh_before_sl.iloc[-1]  # SL直前の最後のSH

Swingパラメータ確定値:
  4H: n=3 / 1H: n=3 / 15M: n=3 / 5M: n=2

4H構造優位性フィルター（#026d確定）:
  neck_4h >= neck_1h（エントリー有効の必要条件）
```

### F-8. 裁量思想の 3 原則 — 原則α / β / γ（2026-04-19 ボス言明・2026-04-20 採番）

**最上位の位置づけ**: 本 ADR は `MINATO_MTF_PHILOSOPHY.md` の第 0 章追記候補として予約
されているボスの裁量思想の 3 原則を、設計方針ガイドとして Trade_System 実装判断に
接続するものである。

**出典**: MTF_INTEGRITY_QA.md 2026-04-19 セッション末尾のボス言明

> 裁量トレードとは相場環境変化による条件反射でロジックが複雑に変化する領域なので、
> いつでも基本に戻れるシンプルな土台をシッカリ作っておくことが何より大切だ。

#### 原則α（最上位・シンプルな土台の保守）

```
裁量トレード = 条件反射でロジックが複雑化する領域
  ↓
対策 = いつでも基本に戻れるシンプルな土台を死守する
  ↓
実装方針 = 拡張は「基本の上に乗せる」形で行い、基本を汚染しない
```

**適用範囲**:
- ロジック設計（D-12/D-13 の創作混入排除）
- ファイルシステム構造（E-8 の src/ 構造再編）
- 運用判断（過剰なフィルター追加の抑制）

**エンジニアリング等価物**: YAGNI 原則・単一責任の原則・オッカムの剃刀と同一の本質。
裁量トレーダーのボスとシニアエンジニアが独立に同じ結論に到達するのは、
人間の認知処理の本質に根差しているから。

**派生原則（2026-04-20 v3 Handoff で追加）**: Trade_Brain / Trade_System 役割分担
- Trade_Brain: 市況データ / トレード結果 / Plot 抽出（静的データ側）
- Trade_System: シグナル / Fibonacci / BackTest（動的ロジック側）
- これはボス 2026-04-20 言明「Trade_Brain は Plot 抽出のみ、シグナル/Fib は Trade_System」
  から導かれる運用境界

**追加派生原則（2026-04-20 Phase 2 で確立）: 共存保持の許容 — plotter.py 両リポ共存判断**

役割分担原則を機械的に適用すれば「機能単位で完全分離」が結論になる。しかし
ボスは 2026-04-20 の plotter.py 処置判断で、異なる結論を出した:

> 互いのルーツがあるというなら両リポのプロジェクト進展で将来補い合う可能性が
> あるので共存させておくのが得策だな。

**共存保持許容の条件**（本派生原則の発動条件）:

1. 対象ファイルが複数ルーツの関数を含む（混在の歴史的経緯が明確）
2. それぞれのルーツ側の呼び出し経路が両リポで完全分離している（相互干渉なし）
3. 将来の合流点として具体的な連携シナリオが想定できる
4. 削除した場合に復元コストが発生する

**plotter.py が満たす条件**:

- 条件 1: ✅ docstring「8 ペア正規化比較プロットおよび Swing High/Low 視覚化
  デバッグ機能」が 8 ペア市況ルーツを明示。その後 #026d BackTest 系 4 関数が
  癒合（混在の歴史的経緯が文書化済み）
- 条件 2: ✅ Trade_Brain 側は `save_normalized_plot` / `save_swing_debug_plot` /
  `save_entry_debug_plot`、Trade_System 側は `plot_base_scan` / `plot_swing_check` /
  `plot_4h_1h_structure` / `plot_1h_window_5m` で呼び出し経路が完全分離
- 条件 3: ✅ ボスが具体的合流シナリオを明言（次項参照）
- 条件 4: ✅ 削除すれば Trade_Brain からのコピーバックが必要

**具体的合流シナリオ（ボス 2026-04-20 言明）**:

> plotter.py というスクリプトが生まれた存在意義があるとしたら、今後 Trade_System
> が進化したとき 30 日変動率の抽出データできる Trade_Brain 側の市況レジーム判定
> を、Trade_System での優位性根拠としてエントリー lot 調整に生かせるという発想
> が生まれた。

**合流シナリオの構造図**:

```
Trade_Brain 側                              Trade_System 側
─────────────────                          ─────────────────
8 ペア 30 日変動率抽出                     エントリー判定（#026d コア）
  (test_fetch_30days_multi)                 (window_scanner.py)
         ↓                                          ↓
save_normalized_plot で可視化              優位性スコア算出
（plotter.py Trade_Brain ルーツ）          （MINATO_MTF_PHILOSOPHY 裁量 STEP ①⑥⑦）
         ↓                                          ↑
市況レジーム判定                        ← ここが合流点（将来）
（Strategy_Wiki/Regimes）                    レジーム情報をロット調整に反映
```

**裁量思想との接続**: この合流シナリオは MINATO_MTF_PHILOSOPHY.md 第 4 章
「将来の段階的導入対象」テーブルの以下 3 項目と直接対応する:

- STEP ①（日足スコア）— マクロ環境認識の自動化
- STEP ⑥（ボラティリティ係数）— 30 日変動率データが直接入力となる
- STEP ⑦（ロット調整）— 優位性スコア × ボラ係数のアウトプット先

つまり plotter.py の両リポ共存は「偶然の癒合」ではなく、**裁量思想に既に
書かれていた未来への伏線**が物理ファイルレベルで橋として機能する構造を持つ。

**「起こるべくして起こる」原則**（ボス 2026-04-20 言明）:

> 物事は起るべきして起こるという発想があれば、ある意味有効的に使えるというものだね。

これは裁量トレーダー視点の洞察だが、エンジニアリング実装判断にも適用できる:
相場で偶然に見えるチャートパターンが構造的必然（フラクタル構造）から生まれる
のと同様に、リポ分離で偶然癒合していた関数が将来の橋渡しとして機能する構造を
予め孕んでいた。

**含意**: 原則α（シンプルな土台の保守）は大事だが、**シンプル化を急ぐあまり
構造的必然を破壊してはいけない**。Evaluator が Phase 1 段階で「plotter.py 関数
分割必須」と書いたのは、この原則を見落としていた反省点である。

**運用ルール**:

- 共存保持判断は Evaluator の机上推論では決定しない（静的シンプル化に偏る傾向）
- ボスの裁量視点（動的エコシステム発展性）を必ず参照する
- 共存保持を採用した場合は、本 ADR のように「具体的合流シナリオ」を文書化する
  （将来の Evaluator / Planner が判断経緯を辿れるように）

#### 原則β（現段階の決済哲学・ノーリスク化後は伸ばさない）

**ボス言明**（2026-04-19）:

> 現時点の実装では 4H SH 到達半値決済時点でノーリスクだから 15Mダウ崩れで良い

```
4H SH 到達半値決済 = ノーリスク状態達成 = 目的完了
  ↓
残り 50% は「15M ダウ崩れ」でシンプルに決済（現段階）
  ↓
建値指値による「4H 3波優位性で伸ばす」ロジックは将来拡張領域
```

**含意**: D-12（建値移動）・D-13（1H 実体確定）などの追加条件は、原則β に反する
創作として退ける。裁量整合版への再設計は Phase 4 で実施する。

```
新機能 A を導入したい

  ↓ No → 導入を待つ（原則γ）
  ↓ Yes → 裁量思想と整合するか？
          ↓ Yes → 実装指示書を Planner 起草
```
