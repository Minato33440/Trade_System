## A. スキャン方向・タイミング系

### A-1. 窓左端スキャンバグ（#021 → #022で修正）

**症状**: 全13件のエントリーが「底を打つ前」に発火していた

**原因**:
`scan_window_entry()` が窓の先頭（1H SL より20時間前）から
5Mバーを順にスキャンし、最初の「min(open,close) > neck_15m」を拾っていた。
これは下落途中でまだネック上にいる足であり、底形成後の上抜けではなかった。

```
価格
 ↑  ①誤検出（下落前）    ③正解（底の後）
 |  ＼  neck -------＼---/------
 |    ＼           ＼ /
 |     ＼___底(SL)__/
 +--------------------------------→ 時間
      窓の左端    1H SL    窓の右端
```

**修正（3行）**:
```python
sl_1h_idx = df_5m_win.index.searchsorted(sl_1h_ts)
for j in range(sl_1h_idx, len(df_5m_win) - 1):
```

**教訓**: 時間窓スキャンでは「どの時点以降からスキャンするか」を必ず明示する。
窓全体からパターンを検出するのは正しいが、エントリー判定は底確定後に限定すべき。

---

### A-2. 1H SL 一致検証で別の波の底を比較（#020旧版 → #020v2で修正）

**症状**: 4H SL と 1H SL の距離が median 206 pips と表示された

**原因**:
`get_all_swing_lows_1h(lookback=240).min()` は10日間の**絶対最安値**を返す。
4H SL が「3日前の押し目底 = 157.5」なのに、1H側は「8日前の大底 = 154.0」を返し、
**別の波の底**同士を比較していた。

**修正**:
4H SL のタイムスタンプ ±8本（8時間）窓内で**最近傍**の1H SLを探す方式に変更。
→ 結果: 89件100%一致（距離0.0pips）

**教訓**: 「最安値」と「同じ構造の安値」は全く別の概念。
時間的近傍で探さないと、異なる相場局面の値を比較してしまう。

---

### A-3. Support_1h の lookback が短すぎた（#020で発覚・#021で転換）

**症状**: 旧版の Support_1h が不正確で、4H構造と噛み合わない

**原因**:
`get_nearest_swing_low_15m(n=3, lookback=20)` → 直近20本 = 5時間分。
4H足の押し目波が形成されるには約1週間かかる。5時間の窓では
4H構造の押し目底を捉えられず、構造的にミスマッチしていた。

**修正**:
3層階層スキャンに転換。4H SL → 1H窓 → 窓内15M/5Mの
トップダウン構造に変更し、lookback依存の問題を根本解消。

**教訓**: lookback の長さは「検出したい構造の形成期間」に合わせる必要がある。
下位足の短い lookback で上位足の構造を捉えようとすると構造的に破綻する。

---

### A-4. 1H SL 選択の後期バイアス — 再テストを初回より優先（#024a分析 → #025で解決）

**症状**: #02（DB 20240726）で本来 Jul26 08:50 がエントリー機会なのに
`sl_1h_ts = Jul26 15:00`（6時間後の再テスト）が選択され機会損失が発生した

**原因**:
`get_1h_window_range()` で `dists.idxmin()`（4H SL 価格に最も近い 1H SL を選択）を使用。
同一価格水準に2回タッチ（初回: 08:50 / 再テスト: 15:00）がある場合、
再テストの方が価格精度が高いため後者が選ばれる。

**修正（#025で完了）**:
```python
# 変更後: 価格許容範囲内で「最も早い(最初の)」1H SLを優先
price_tol_pips = 20.0
close_enough   = sl_1h_near[dists <= price_tol_pips * PIP]
sl_1h_ts = close_enough.index[0] if len(close_enough) > 0 else dists.idxmin()
```

**教訓**: 「最も価格が近い構造点」と「最初に形成された構造点」は異なる概念。
押し目底は初回タッチが本来の機会であり、再テストは追認にすぎない。

### A-5. neck 選択の後期バイアス — 最高値SHが本来のneckより上を指す（#024a分析 → #025で解決）

**症状**: #03（DB 20240729）で本来のneckは153.692付近なのに
`neck_15m = 153.913`（1段上のレンジ）が選択されレイトエントリー（約22pips遅れ）

**原因**:
`scan_window_entry()` で `sh_vals.max()`（1H SL以降の15M SH最高値）を使用。

**修正（#025で完了）**:
```python
neck_15m = float(sh_vals.iloc[0])  # 固定ネック原則
```

**結果**: 4件 → 15件（+275%）

**教訓**: 「最高値」が「最初の抵抗線」を上回るのは、その抵抗線をすでにブレイクした後。
最初の抵抗線がneckであり、ブレイク後の高値はもはやneckではない。

## B. API シグネチャ不一致系

### B-1. get_direction_4h の current_idx 問題（#020）

**症状**: ClaudeCode が指示書通りに呼び出すと TypeError

**原因**:
指示書では `get_direction_4h(high, low, n=3, lookback=20)` と想定していたが、
実際は `get_direction_4h(high, low, current_idx, n=3, lookback=20)` で
`current_idx` が必須引数だった。

**教訓**: 指示書のコードは「想定」であり、実際のAPIと一致しない可能性が常にある。
ClaudeCode には必ず「実ファイルの def 行を read してから呼び出すこと」と指示する。

---

### B-2. check_15m_range_low の返り値型（#021）

**症状**: `result[0]`, `result[1]` でアクセスすると KeyError

**原因**:
実際は `→ dict` で `result['found']`, `result['pattern']`, `result['neck_15m']` だった。

**#025 最終確定**: check_15m_range_low() はパターンラベル専用。
neck の計算は独立（固定ネック原則 = sh_vals.iloc[0]）。

---

### B-3. 指示書に旧版と修正版が混在（#020指示書）

**症状**: ClaudeCode が旧版（バグあり）を実装してしまうリスク

### E-6. neck_1h / neck_4h の取得方法 — CSV追加 vs exit内再計算（#026設計判断）

**問題**:
#026（manage_exit統合）で、position dict に必要な `neck_1h`（半値決済トリガー）
と `neck_4h`（段階3移行トリガー）が window_scan_entries.csv に存在しない。

**選択肢A（却下）**: exit_simulator 内で再計算
```
exit_simulator.py が swing_detector の関数を呼び、
entry_ts 時点の 4H SH / 1H SH を都度計算する。
```

**選択肢B（採用）**: window_scanner.py に sh_4h / neck_1h カラムを追加
```
scan_4h_events() で 4H SH も取得し CSV に保存。
get_1h_window_range() で窓内 1H SH も取得し CSV に保存。
```

**Bを採用した理由（Evaluator判断）**:

1. **構造データの補完**: 4H上昇ダウの構造上、各4H SLには対応する4H SHが
   必ず存在する。SLだけ保存してSHを保存しないのは「構造情報の半分しかない」
   状態であり、設計の欠落。neck_1h も同様。

2. **DRY原則**: exit_simulator で再計算すると、window_scanner と
   同じ計算を別々に行うことになる。計算結果が微妙にズレるリスクがある。

3. **エントリー文脈情報はエントリー時に確定すべき**: neck_1h / neck_4h は
   エントリーが成立した時点の市場構造の一部。決済時に「後から再計算」するのは
   設計として不適切。エントリーの文脈は CSV に全て含まれるべき。

4. **exit_simulator の自己完結性**: CSV から全情報を読み取り →
   manage_exit() を呼ぶだけのシンプルな構造にできる。
   swing_detector への依存が不要。

**window_scanner.py 変更の正当性**:
window_scanner.py は #025 で「完了」だが「凍結」ではない。
凍結ファイル（backtest.py / entry_logic.py / exit_logic.py / swing_detector.py）
とは異なり、機能拡張が許可されている。
（2026-04-13 ボス承認済み）

**半値決済の執行価格**: α方式（neck_1h 固定値）を採用。
理由: 戦略定義が「4Hネックライン到達で半値決済」であり、
指値注文と同等。Close 価格（β方式）はスリッページが不必要に大きくなる。

**実装順序（確定）**:
```
#026a: window_scanner.py に sh_4h / neck_1h カラム追加
       → CSV再生成（15件 + 新カラム）
#026b: exit_simulator.py 新規作成
       → CSV読み込み → position dict構築 → manage_exit() → P&L算出
```

## F. 設計方針ガイド（Planner必読）

**本章は Evaluator（Opus）が発行・管理する。**
Planner が設計判断で迷った際の判断基準として使うこと。
方針に沿った選択肢を採用し、逸脱する場合は Evaluator に確認を取ること。

---

### F-1. データの流れ: 上位足 → 下位足（トップダウン原則）

```
確定済みの3層構造:
  Layer 1（4H）→ Layer 2（1H窓）→ Layer 3（15M/5M窓内）

全ての情報は上位足から下位足へ渡す。
下位足が上位足の情報を「逆算」や「再計算」してはならない。

具体例:
  ✅ 4H SH/SL → 1H窓確定 → 15Mパターン検出 → 5Mエントリー
  ❌ 5Mエントリー後に exit_simulator が 4H SH を自前で再計算
```

**適用場面**: 新しいカラムや情報が必要になった場合、
「どのレイヤーで確定すべきか」を考える。
答えは常に「その情報が構造的に属するレイヤー」。

---

### F-2. エントリー文脈はエントリー時に確定する

```
エントリーが成立した時点で、そのトレードに関する全ての構造情報を確定し、
CSV（または同等の永続化手段）に保存する。

確定すべきエントリー文脈:
  ✅ entry_price, entry_ts        （エントリー自体）
  ✅ sl_4h, sh_4h                 （4H構造 = 損切/利確の基準）
  ✅ neck_15m                     （15Mエントリートリガー）
  ✅ neck_1h                      （半値決済トリガー）
  ✅ pattern                      （DB/IHS/ASCENDING）
  ✅ sl_1h_ts, sl_4h_ts           （構造の時間的位置）

決済ロジックは CSV から読み取るだけ。再計算しない。
```

**理由**: 決済時に再計算すると、lookback位置やデータ範囲の違いで
エントリー時と異なる値が出るリスクがある。
一度確定した構造情報は不変であるべき。

---

### F-3. 関数の責務を単一にする（用途限定の原則）

```
一つの関数に複数の役割を持たせない。

確定済みの用途限定:
  check_15m_range_low() → パターンラベル取得のみ（#024a確定）
  neck計算             → 1H SL以降の初回15M SH（独立・#025確定）
  manage_exit()        → 決済判定のみ（position dictを受け取る側）
  scan_4h_events()     → 4H構造スキャン（SH/SLペアで返す）

新機能を追加する場合:
  ✅ 既存関数の「返り値を拡張」（例: sh_4h も返す）
  ❌ 既存関数の「内部ロジックを変更」
  ❌ 一つの関数で検出+計算+判定を同時に行う
```

**理由**: 関数の責務が複合していると、バグの切り分けが困難になる。
#021（check_15m_range_low がパターン+neck を同時に担当）で
neck が不正確なのにパターンのせいだと誤認した事例がある。

### F-4. ファイル変更ポリシー（2026-04-13 更新）

```
■ 凍結ファイル（変更禁止・明示的許可があっても慎重に）
  - backtest.py        （#018ベースライン保持）
  - entry_logic.py     （#018まで凍結）
  - exit_logic.py      （#009以降凍結）
  - swing_detector.py  （#020まで凍結）

  → これらは比較のベースラインや、他ファイルが依存するAPIを提供している。
    変更するとベースライン数値が再現不能になるか、
    依存先が壊れるリスクがある。

■ 拡張可能ファイル（機能追加OK・ロジック変更は要確認）
  - window_scanner.py  （#025完了だが凍結ではない）
  - plotter.py         （表示機能の追加は自由）
  - structure_plotter.py

  → カラム追加、新しい出力、表示改善などは Planner 判断で可。
    既存のスキャンロジックやパターン検出ロジックの変更は
    Evaluator 確認を取ること。

■ 新規作成（自由）
  - exit_simulator.py  （#026b で作成予定）
  - その他の独立スクリプト

  → 既存ファイルを import して使うのは OK。
    既存ファイルの内部を書き換えるのは NG。
```

**判断基準**: 「このファイルを変更したら、過去の数値（#018ベースライン等）が
再現できなくなるか？」→ Yes なら凍結。No なら拡張可能。

### F-5. 設計判断で迷った時の優先順位

```
迷った時は以下の順で判断する:

1. 上位足構造の整合性 > 実装の簡便さ
   例: neck_1h が無い → CSV追加（構造補完）> exit内で再計算（楽だが歪み）

2. エントリー時確定 > 決済時再計算
   例: sh_4h が必要 → scan_4h_events で取得 > exit_simulator で再計算

3. シンプル版を先に > フル版を一気に
   例: Phase 1（15Mネック越え）→ Phase 2（15M右肩内5M DB）

4. バグ切り分け可能 > 機能の網羅性
   例: 1つの指示書で1つの変更 > 3つの変更を同時に

5. 過去の失敗パターン（A〜D章）に該当しないか確認
   特に: スキャン開始位置（A-1）、後期バイアス（A-4/A-5）、
   API不一致（B-1/B-2）は繰り返しやすい
```
